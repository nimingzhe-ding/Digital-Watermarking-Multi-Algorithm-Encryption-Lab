# 数字水印与多算法加密实验平台项目功能实现原理详解

## 1. 项目背景与目标

我们这个项目的目标，是搭建一个能够在线演示“数字水印 + 数字加密”相关功能的交互式网站。和普通静态展示网页不同，我们希望它既能真正处理图片和文件，又能体现一定的安全性、实验性和可操作性。

从课程项目的角度来看，我们主要解决了下面几个问题：

1. 如何把水印嵌入到图片中，而且尽量不让人直接看出来。
2. 如何提供多种加密算法，方便用户根据需要选择。
3. 如何把文本、文件、图片处理这些不同能力统一到一个网站里。
4. 如何把实验结果保存下来，形成一个可管理的“用户上传作品库”。
5. 如何让整个网站不仅能用，而且看起来更像一个完整的实验平台。

因此，我们最终把项目做成了一个多模块网站，首页先选择功能，再进入对应模块页面，分别完成不可见水印、可见水印、多算法加密以及作品归档管理等任务。

---

## 2. 项目整体架构

整个项目采用的是“前后端分离但部署在同一个 Node 服务中”的实现思路。

### 2.1 架构划分

- 前端部分：
  使用原生 `HTML + CSS + JavaScript` 实现页面展示与交互。
- 后端部分：
  使用 `Node.js + Express` 负责接口、文件上传、加密解密、静态资源服务。
- 图像隐写部分：
  使用 `Python + Pillow` 单独完成不可见水印的嵌入、提取和容量估算。

### 2.2 为什么采用 Node + Python 混合方案

我们在设计时并没有强行把所有内容都写在一种语言里，而是根据功能特性来分工：

- `Node.js` 更适合做 Web 服务、接口组织、文件上传下载、前后端联动。
- `Python` 更适合做图像级别的数据处理，尤其是实验性质的像素操作。

这样做的好处是：

- 后端接口层更清晰。
- 图像水印算法可以独立调试。
- 后期如果要替换为更复杂的隐写算法，也比较容易扩展。

这也是我们这个项目比较有工程感的一点，不是简单地把所有功能堆在一个文件里。

### 2.3 项目目录结构

```text
.
├─ public/
│  ├─ index.html
│  ├─ invisible.html
│  ├─ visible.html
│  ├─ crypto.html
│  ├─ library.html
│  ├─ styles.css
│  └─ app.js
├─ python_tools/
│  └─ invisible_watermark.py
├─ storage/
│  └─ library/
│     ├─ files/
│     └─ index.json
├─ server.js
├─ package.json
├─ requirements.txt
├─ README.md
└─ 项目功能实现原理详解.md
```

其中：

- `public/` 负责前端页面和交互逻辑。
- `server.js` 是后端主程序。
- `python_tools/invisible_watermark.py` 是不可见水印核心算法。
- `storage/library/` 是我们新增的本地作品库。

## 3. 技术选型与原因

### 3.1 前端

前端使用的是原生三件套：

- `HTML`：负责页面结构
- `CSS`：负责科技感视觉风格
- `JavaScript`：负责表单提交、接口请求、结果回显、预览与下载

我们没有使用 Vue 或 React，一方面是为了降低部署复杂度，另一方面是课程项目中原生前端更容易看清楚每个功能的真实实现过程。

### 3.2 后端

后端基于以下依赖：

- `Express`
  用于快速搭建接口服务
- `Multer`
  用于处理文件上传
- `Sharp`
  用于进行图片合成和可见水印处理
- Node 内置 `crypto`
  用于实现多种加密算法

### 3.3 Python 图像处理

不可见水印部分使用：

- `Pillow`
  用于读取图像、转为 RGBA 像素数组、重新保存 PNG

这个部分我们采用 Python 的原因是，按字节和像素处理图像时，Python 写起来更直观，更适合实验型算法验证。

## 5. 前端页面设计思路

### 5.1 首页分发式结构

我们没有把所有功能都堆在一个页面里，而是采用“首页选功能，进入对应模块页面”的设计。

这样做有三个原因：

1. 功能边界更清楚，不容易混乱。
2. 不同模块的表单和逻辑不会互相干扰。
3. 更符合实际产品的导航习惯，也更适合答辩展示。

### 5.2 科技感视觉设计

前端整体风格偏“实验室 / 数字安全平台”：

- 深蓝色背景
- 青色、绿色、橙色的光晕叠加
- 网格化背景
- 半透明玻璃卡片
- 圆角面板 + 发光边框

这种风格并不是随便换个颜色，而是为了让整个网站更贴合“数字水印”“加密”“实验平台”这个主题。

### 5.3 页面初始化方式

前端的 `app.js` 并不是所有页面都执行同一套逻辑，而是通过判断当前页面是否存在某个表单，再初始化对应模块。

例如：

- `initInvisiblePage()` 只负责不可见水印页
- `initVisiblePage()` 只负责可见水印页
- `initCryptoPage()` 只负责加密页
- `initLibraryPage()` 只负责作品库页

这样做的优点是：

- 每个页面职责清晰
- 减少无效事件绑定
- 避免一个页面引用另一个页面不存在的 DOM 元素时报错

## 6. 不可见水印模块的实现原理

这一部分是我们项目中最有技术含量的内容之一。

### 6.1 基本思想

我们采用的是基于 `LSB` 的隐写思路，也就是最低有效位写入。

简单来说，一张彩色图片的每个像素可以拆成多个通道值，例如：

```text
R = 10110010
G = 01101101
B = 11100110
```

如果我们只修改最低位，比如把 `11100110` 改成 `11100111`，人眼通常很难察觉颜色变化。于是我们就可以把二进制信息一点一点藏进图片像素里。

### 6.2 为什么能做到“看不见”

原因在于：

- 人眼对单个像素最低位变化不敏感
- 改动幅度通常只有 `+1` 或 `-1`
- 写入信息分散在整张图中，不会形成肉眼明显的结构

因此，从视觉上看，处理前后的图片几乎一样，但实际上图片内部已经携带了隐藏信息。

### 6.3 不可见水印写入的数据内容

我们并不是只写一个简单字符串，而是先构造一个结构化的水印内容对象，大致包括：

- `watermarkText`
- `author`
- `owner`
- `note`
- `createdAt`

如果用户设置了密码，那么这部分元数据会先加密，再嵌入图片；如果没有设置密码，就直接以 JSON 的形式嵌入。

### 6.4 水印头部设计

为了让提取过程知道“这张图里是否真的有我们定义的水印，以及水印长度是多少”，我们设计了固定长度的头部。

头部长度为 `16` 字节，包含：

- 魔数：`IWM1`
- 版本号
- 通道模式
- 重复写入次数
- 负载长度
- CRC32 校验值

这样做的作用是：

1. 提取时先判断图片中是否存在我们定义的水印格式。
2. 知道后面的有效数据应该读取多少字节。
3. 用 CRC32 检验提取结果是否被压缩、裁剪或篡改。

### 6.5 通道模式设计

项目支持 4 种通道模式：

- `rgb`
- `red`
- `green`
- `blue`

含义是水印正文写入哪些颜色通道的最低位。

不同模式的特点：

- `rgb`
  可用容量更大，但写入更分散
- `blue`
  蓝色通道较适合隐写，常见实验中使用较多
- `red` / `green`
  便于做通道对比实验

### 6.6 重复写入与鲁棒性思路

项目支持 3 种重复模式：

- `1`
- `3`
- `5`

其核心思想是：

- 如果重复次数是 `1`，每一位只写一次，容量最大
- 如果重复次数是 `3` 或 `5`，同一个逻辑位会被重复写入多个位置

在提取时，我们采用“多数投票”的方式恢复位值，例如重复 `3` 次时：

```text
0, 1, 1 -> 提取结果为 1
0, 0, 1 -> 提取结果为 0
```

这样做的意义是：

- 提高一定的抗微小扰动能力
- 在少数位被破坏时仍有机会恢复原始信息

当然，这种鲁棒性仍然是有限的，无法抵抗严重压缩和大幅裁剪，但作为课程实验已经足够体现思路。

### 6.7 容量估算原理

在真正嵌入之前，我们提供了“容量估算”功能，用于判断当前图片在当前模式下最多还能写入多少信息。

容量估算的核心逻辑是：

```text
可用逻辑位数 = 可写入位数 / repetition
可用字节数 = 可用逻辑位数 / 8
```

其中：

- 可写入位数取决于总像素数量
- 通道模式不同，可用位置不同
- 重复次数越高，可写的实际内容越少

这个功能的意义在于，用户不用等到真正嵌入时报错，而是可以先看到“容量是否足够”。

### 6.8 头部与正文写入位置

我们在实现中做了一个比较清晰的分工：

- 水印头部固定写到前一段像素的蓝色通道最低位中
- 水印正文再根据通道模式写入其他位置

这么做的优点是：

- 提取头部时不需要猜测位置
- 格式识别更加稳定
- 有利于后续解析正文长度与参数

### 6.9 CRC32 校验

提取时，除了把数据读出来，我们还会重新计算 CRC32，与头部中保存的值比较。

如果不一致，就说明：

- 图片可能被修改过
- 图片可能被压缩过
- 图片可能被裁剪了
- 数据已经损坏

于是系统会提示用户水印校验失败，而不是返回错误数据。

### 6.10 PSNR 指标

为了体现实验过程的可视化和学术性，我们在嵌入后还计算了 `PSNR`。

`PSNR` 全称是峰值信噪比，常用于衡量图像处理前后差异程度。

公式可写为：

```text
PSNR = 20 * log10(255 / sqrt(MSE))
```

其中 `MSE` 是均方误差。

PSNR 越高，说明改动越小，图像越接近原图。

这个指标在课程展示中很有用，因为它能从数字上说明“虽然嵌入了水印，但图像质量变化很小”。

### 6.11 为什么输出 PNG

不可见水印嵌入后我们统一输出为 `PNG`，主要原因是：

- PNG 是无损格式
- 不会像 JPEG 那样自动压缩像素值
- 更适合保留最低位信息

如果嵌入完又转成 JPEG，很多最低位信息会被压缩破坏，导致水印无法提取。

所以我们在系统中也专门提示用户尽量保存原始 PNG 结果图。

## 7. 不可见水印模块的后端流程

以“嵌入不可见水印”为例，整体流程如下：

```text
前端表单提交
-> Express 接收图片和参数
-> 组装水印 JSON 元数据
-> 如果设置密码，先对元数据加密
-> 把结果写成 payload.bin
-> Node 调用 Python 脚本
-> Python 读取图片像素并进行 LSB 写入
-> 生成新的 PNG 图片
-> Python 返回指标信息
-> Node 把图片和指标返回给前端
-> 前端展示预览并提供下载
```

而提取流程是：

```text
前端上传带水印图片
-> Node 调用 Python 提取 payload
-> Python 返回 base64 编码的隐藏数据
-> Node 解析水印包
-> 若水印是加密的，则提示输入密码
-> 密码正确后解密 JSON
-> 前端显示最终提取结果
```

这套流程说明我们不是把所有逻辑写死在前端，而是把“像素级处理”和“业务级解析”做了职责分离。

## 8. 可见水印模块的实现原理

可见水印模块虽然原理上没有不可见水印复杂，但它承担了很重要的对照作用。

### 8.1 基本思路

这个模块不是修改像素最低位，而是直接把带文字的矢量层叠加到原图上。

我们用 `Sharp` 处理原图，同时动态生成一段 SVG 水印图层，再把它覆盖到原图之上。

### 8.2 为什么使用 SVG 叠加

因为 SVG 有几个优点：

- 文本可以自由设置颜色、大小、旋转角度
- 可生成单个水印或斜向平铺水印
- 不需要额外引入复杂字体渲染方案
- 与 `Sharp` 合成方便

### 8.3 支持的参数

可见水印支持：

- 文本内容
- 水印颜色
- 透明度
- 字号比例
- 旋转角度
- 位置模式

其中“斜向平铺”模式会在整张图上循环铺设水印，更接近实际版权标注场景。

### 8.4 可见水印的意义

这个模块虽然不是项目核心创新点，但在课程展示里非常有意义：

- 可以和不可见水印做直观对比
- 更容易向老师解释“可见水印和不可见水印的区别”
- 让系统功能更完整，不会只停留在一个实验点上

## 9. 多算法加密模块的实现原理

这一部分主要解决“加密方式单一”的问题。

### 9.1 为什么要做多算法

如果系统只支持一种加密算法，会显得功能比较单薄，也不方便做对比实验。

所以我们把它扩展成了一个“多算法加密工具箱”，让用户在加密时可以手动选择算法，在解密时系统自动识别算法。

这更符合课程项目里“可对比、可展示、可分析”的要求。

### 9.2 支持的算法分类

目前系统中的算法大体分为两类：

#### 9.2.1 AEAD 类算法

- `AES-128-GCM`
- `AES-192-GCM`
- `AES-256-GCM`
- `ChaCha20-Poly1305`

这类算法的特点是：

- 同时提供加密和认证
- 解密时能判断密文是否被篡改
- 结构相对现代，安全性更完整

#### 9.2.2 “加密 + HMAC”组合算法

- `AES-256-CBC + HMAC-SHA256`
- `AES-256-CTR + HMAC-SHA256`

这类算法的特点是：

- 先用分组或流模式进行加密
- 再用 `HMAC-SHA256` 做完整性校验

也就是说，它们虽然不是 AEAD，但通过“加密 + MAC”的组合，也能实现“既保密又防篡改”的效果。

### 9.3 密钥派生方式

系统中的口令并不是直接拿来做加密密钥，而是先通过：

```text
PBKDF2-SHA256 x 150000
```

派生出真正的密钥。

这样做的原因是：

- 用户输入的密码长度不固定、强度不稳定
- 直接拿密码做密钥不安全
- 通过 `salt + PBKDF2` 可以提升口令攻击成本

我们还为每次加密都随机生成 `salt`，从而避免相同密码每次产生相同密文结构的问题。

### 9.4 统一密文封装格式

为了让不同算法的密文都能被系统统一处理，我们设计了自己的密文包格式。

新版格式使用魔数：

```text
DWE2
```

后面再跟上：

- 算法 ID
- salt 长度
- iv 长度
- tag 长度
- salt
- iv
- tag
- encrypted payload

这样解密时就不需要用户告诉系统“我刚才用的是哪种算法”，因为系统会先读取密文头部，再自动判断使用哪一种算法去解密。

### 9.5 为什么还保留旧格式兼容

项目早期只有一种加密方案时，系统已经生成过一批旧密文，我们把它们称为：

```text
DWE1
```

如果后来升级算法后就完全不兼容旧格式，那么之前的数据就无法解密了。

所以我们在后端保留了旧版兼容逻辑，做到：

- 新数据用 `DWE2`
- 旧数据仍能解密

这体现了我们在项目设计中考虑了“版本兼容性”。

### 9.6 AEAD 模式的处理方式

对于 `AES-GCM` 和 `ChaCha20-Poly1305`，我们使用的是带认证标签的加密流程。

大致过程：

1. 根据密码派生密钥
2. 随机生成 `iv`
3. 用算法对明文加密
4. 取出认证标签 `tag`
5. 把头部参数与密文一起封装

同时我们把前缀信息作为 `AAD` 使用，让算法在认证时也把格式头部纳入保护范围，避免攻击者伪造算法描述字段。

### 9.7 CBC / CTR + HMAC 的处理方式

对于这两种方案，我们不是只做加密，而是把派生出来的密钥材料拆成两部分：

- 前 32 字节：加密密钥
- 后 32 字节：HMAC 密钥

然后：

1. 用加密密钥进行加密
2. 用 HMAC 密钥对“头部 + salt + iv + 密文”做完整性校验
3. 解密前先验证 HMAC

这样可以避免出现“能解密但无法判断是否被篡改”的问题。

### 9.8 文本加密流程

文本加密的流程较简单：

```text
输入明文
-> 选择算法
-> 输入密码
-> 后端加密
-> 返回 Base64 形式的密文包
```

之所以最终显示为 Base64，是因为：

- 二进制密文不适合直接放在文本框里
- Base64 更方便复制、粘贴和展示

### 9.9 文件加密流程

文件加密比文本复杂一点，因为除了内容本身，我们还要保留原始文件名和 MIME 类型。

因此我们先把文件封装成：

```text
[4字节 metadata长度] + [metadata JSON] + [文件内容]
```

其中 `metadata JSON` 里面包含：

- `filename`
- `mimeType`

然后再把整个二进制包进行加密。

解密后，系统就可以恢复：

- 原始文件内容
- 原始文件名
- 原始文件类型

这也是为什么解密下载下来的文件能尽量接近原始状态。

### 9.10 水印模块与加密模块的结合

不可见水印中的“密码保护”其实就是在嵌入前调用了同一套加密逻辑。

也就是说：

- 用户不设置密码：直接嵌入 JSON
- 用户设置密码：先按所选算法加密 JSON，再把密文嵌入图片

这样我们的系统并不是“水印模块”和“加密模块”完全割裂，而是做了真正的功能融合。

## 10. 作品库模块的实现原理

这是我们后期新增的一个比较实用的模块。

### 10.1 设计目标

前面的几个模块虽然都能生成结果，但如果用户每次处理完文件就直接下载到本地，网站内部其实没有“实验成果沉淀”。

所以我们增加了“用户上传作品库”，用于：

- 统一保存实验结果
- 方便答辩展示
- 方便后续管理和复用

### 10.2 为什么没有直接上数据库

我们这里采用的是“本地文件系统 + JSON 索引”的方案，而不是数据库。

原因主要有三个：

1. 课程项目部署更简单。
2. 当前数据量不大，本地文件足够支撑。
3. 可以先把业务流程跑通，后期再升级成 SQLite 或 MySQL。

这种做法很适合课程项目阶段，既能完整体现功能，又不会把精力过多消耗在数据库配置上。

### 10.3 存储结构

作品库存储在：

```text
storage/library/files/
storage/library/index.json
```

其中：

- `files/` 存放原始作品文件
- `index.json` 存放每个作品的元数据

元数据大致包括：

- `id`
- `title`
- `creator`
- `category`
- `description`
- `tags`
- `originalName`
- `storedName`
- `mimeType`
- `size`
- `createdAt`

### 10.4 上传流程

用户上传作品时，后端会执行以下步骤：

1. 接收上传文件
2. 对原始文件名做安全处理
3. 生成唯一 ID
4. 生成实际存储文件名
5. 保存文件到 `files/`
6. 把元数据追加写入 `index.json`
7. 返回新作品信息给前端

其中唯一 ID 使用随机 UUID 生成，这样可以避免文件重名冲突。

### 10.5 预览与下载

为了兼顾图像预览和原件下载，我们设计了两个接口：

- `content`
  用于在线查看
- `download`
  用于按附件方式下载

如果作品本身是图片，前端就直接把 `content` 地址放到 `<img>` 里，实现在线预览。

如果不是图片，例如 PDF、压缩包或加密文件，页面就显示文件类型占位卡片。

### 10.6 搜索与筛选

目前作品库的搜索和筛选主要在前端完成。

前端会根据以下字段构造搜索文本：

- 标题
- 作者
- 分类
- 描述
- 原文件名
- 标签

用户输入关键词后，前端直接对当前列表进行过滤。

这个实现方式的优点是：

- 简单直接
- 响应快
- 对课程项目足够实用

如果后续作品数量变大，再改成服务端检索会更合理。

### 10.7 删除机制

删除作品时，后端并不是只删掉索引记录，而是同时：

- 删除 `index.json` 中的那条元数据
- 删除 `files/` 中真正的原始文件

这样可以保证作品库数据一致，不会出现“列表里没了但文件还在”的情况。

## 11. 前后端交互机制

### 11.1 文件上传方式

涉及图片和文件的模块，基本都使用 `FormData + multipart/form-data` 方式提交。

原因是：

- 适合上传文件
- 可以同时提交文本字段和二进制文件
- 与 `Multer` 配合方便

### 11.2 JSON 接口方式

纯文本加密 / 解密使用的是 JSON 请求：

```json
{
  "text": "...",
  "password": "...",
  "algorithm": "aes-256-gcm"
}
```

这种方式更适合纯文本数据，也更方便浏览器直接调用。

### 11.3 自定义响应头

在不可见水印和加密文件下载这些接口中，我们还利用了自定义响应头传递额外信息，例如：

- `X-IWM-Width`
- `X-IWM-Height`
- `X-IWM-Channel-Mode`
- `X-IWM-Repetition`
- `X-IWM-PSNR`
- `X-Encryption-Algorithm`
- `X-Encryption-KDF`

这样前端在拿到文件结果时，还能同步拿到技术指标，并展示在页面上。

这比把所有信息都塞进文件名里更规范，也更方便扩展。

## 12. 错误处理与安全性考虑

### 12.1 文件大小限制

后端上传限制为 `20MB`，这样可以避免：

- 超大文件导致内存压力过大
- 恶意上传占用服务器资源

### 12.2 文件名安全处理

我们对文件名进行了过滤，去除了：

- 路径分隔符
- 控制字符
- 一些非法字符

这样可以避免路径穿越和特殊文件名带来的问题。

### 12.3 解密失败处理

无论是文本解密、文件解密还是加密水印提取，只要密码错误或数据损坏，系统都会返回明确错误，而不会输出错误内容。

### 12.4 全局错误中间件

后端最后统一挂载了错误处理中间件：

- 参数错误返回 `400`
- 找不到资源返回 `404`
- 服务内部异常返回 `500`

这样整个系统的错误输出会更一致，也便于调试和演示。
